# Personal Site

[Live Site](https://www.kushagrabharti.com)

This repository powers two connected products in one codebase:

- `portfolio`: the public portfolio site, public API, `/ai`, and generated `llms.txt`/`portfolio.json`/`version.json`
- `tracker`: a private task and calendar-connected planning app

The backend owns portfolio content and live integrations. The frontend owns rendering, interaction, routing, and the private tracker experience.

![Personal Site Preview](personal-site-preview.png)

## Product Surfaces

### Portfolio

The public site lives at `/`. It statically prerenders the real React homepage shell at build time, hydrates that same shell in the browser, then loads media-heavy enhancements, the 3D hero model, and live data after the first paint without skeleton placeholders.

Machine-readable portfolio surfaces:

- `/ai` renders a structured AI-readable portfolio page.
- `/llms.txt` is generated from the backend portfolio snapshot.
- `/portfolio.json` is generated from the same backend-owned portfolio snapshot for structured crawlers and agents.
- `/version.json` is generated with build/export metadata so stale crawler responses can be diagnosed.
- `frontend/index.html`, `frontend/ai.html`, `frontend/public/llms.txt`, `frontend/public/portfolio.json`, `frontend/public/version.json`, `frontend/public/robots.txt`, `frontend/public/sitemap.xml`, `frontend/src/portfolio/generated/introBootstrap.ts`, and `frontend/src/portfolio/generated/portfolioSnapshotBootstrap.ts` are generated by `frontend/scripts/sync-portfolio-exports.mjs`.

Portfolio content starts in `backend/src/portfolio/content`. Writings/thoughts are configured in `backend/src/portfolio/content/writings.ts`, with hover-preview bodies stored as Markdown in `backend/src/portfolio/content/writings/*.md`. The frontend should render that data instead of becoming a second content source. Public portfolio fonts are self-hosted under `frontend/public/portfolio/fonts` and should not be reintroduced through Google Fonts.

Public portfolio images should render through `PortfolioImage` when possible so browsers prefer optimized AVIF/WebP variants while original PNG/SVG files remain fallbacks.

### Tracker

The private tracker lives at `/tracker`. It is lazy-loaded, Supabase-authenticated, and currently has one active module:

- `tasks`: the current tasks-hub interface

Most tracker CRUD talks directly to Supabase from the frontend. Backend private APIs handle service-role operations, task-list deletion, cron, and Google Calendar sync.

## Getting Started

### Prerequisites

- Bun
- GitHub token for the public GitHub stats API
- OpenWeather API key for the public weather API
- Supabase credentials for tracker auth/data
- Docker Desktop if running the local Supabase stack
- Google Calendar credentials if working on calendar sync

### Frontend

```bash
cd frontend
bun install
cp .env.local.example .env.local
bun run dev
```

Common frontend env:

```bash
VITE_API_BASE_URL=http://localhost:5000
VITE_SUPABASE_URL=
VITE_SUPABASE_ANON_KEY=
```

Only anon/publishable Supabase keys belong in frontend env.

### Backend

```bash
cd backend
bun install
cp .env.example .env
bun run dev
```

Key backend env:

```bash
GITHUB_USERNAME=
GITHUB_TOKEN=
OPENWEATHER_API_KEY=
SUPABASE_URL=
SUPABASE_SERVICE_ROLE_KEY=
```

Backend tracker routes prefer a Supabase service-role JWT in `SUPABASE_SERVICE_ROLE_KEY`, with `SUPABASE_SECRET_KEY` as a fallback server key. Do not use the anon key for backend private APIs.

### Supabase CLI

The repo uses the official Supabase CLI for database migrations, local Supabase, schema diffs, remote pushes, and generated types. Supabase MCP is useful for AI-tool access to a Supabase project, but the CLI is the project-level source of truth because it creates checked-in, reproducible migrations.

```bash
npm install
npm run supabase:link -- --project-ref <your-project-ref>
npm run supabase:start
```

New runnable database changes should go in `supabase/migrations`. The existing `backend/sql` files are schema snapshots and advisor cleanup context unless a file is explicitly written as a migration.

## Common Commands

Frontend:

```bash
cd frontend
bun run dev
bun run build
bun run lint
bun run test:unit
bun run test:integration
```

Backend:

```bash
cd backend
bun run dev
bun run build
bun run lint
bun run test:unit
bun run test:integration
```

Repo-level verification:

```bash
bun install
bun run verify
```

Supabase:

```bash
npm run supabase:status
npm run supabase:migration:new -- migration_name
npm run supabase:db:diff -- --file migration_name
npm run supabase:db:reset
npm run supabase:db:push
npm run supabase:types:local
npm run supabase:types:linked
```

Verification tiers:

- `bun run verify` -> npm-backed installs, then build, lint, unit, integration
- `bun run verify:live` -> `verify` plus live backend checks

The repo is Bun-first for normal app work. The root `verify` scripts intentionally use npm internally because the test toolchain has been more reliable that way in the Windows + OneDrive environment.

## Development Notes

- Preserve the portfolio/tracker separation in both frontend and backend.
- Keep public portfolio asset paths rooted at `/portfolio/...`.
- Use slug-based, ordered portfolio records.
- Remove placeholder links instead of leaving `#`.
- Weather must stay backend-driven and must not trigger browser location prompts.
- GitHub stats should prefer the GraphQL contribution path.
- Finance has been removed from the tracker. Do not reintroduce finance modules, routes, or docs.
- `AGENTS.md`, `CLAUDE.md`, and this `README.md` are the only root docs intended to remain.
